## 1. 과제 개요

### 1.1 과제 목표

본 과제는 **AI 기반 개인 맞춤형 수학 학습 지원 시스템**을 개발하여, 각 학생의 학습 맥락, 능력, 선호도를 종합적으로 분석하고 실시간으로 최적화된 학습 전략을 제공하는 것을 목표로 합니다.

### 1.2 핵심 가치 제안

- **실시간 개인화**: 학생의 실시간 학습 상태를 분석하여 즉각적인 맞춤형 전략 제공
- **주기적 재계획**: 30분 주기 하트비트 시스템을 통한 지속적인 학습 전략 최적화
- **자가개선**: 에이전트 시스템의 자율적 성능 개선 및 진화
- **인간-루프 협업**: 교사의 전문성을 AI 시스템과 결합한 하이브리드 접근

### 1.3 시스템 규모

- **에이전트 수**: 22개 전문 에이전트
- **데이터베이스 테이블**: 200개 이상
- **온톨로지 클래스**: 100개 이상
- **룰 엔진 규칙**: 1,000개 이상
- **지원 플랫폼**: Moodle 3.7 기반

## 2. 연구 목적 및 필요성

### 2.1 연구 배경

현재 교육 현장에서는 다음과 같은 문제점들이 존재합니다:

1. **일괄 교육의 한계**: 동일한 교육 내용과 속도로 모든 학생을 가르치는 방식의 비효율성
2. **개인차 미반영**: 학생의 학습 스타일, 능력, 감정 상태를 고려하지 못하는 문제
3. **사후 대응**: 학습 문제가 발생한 후에야 대응하는 반응형 접근의 한계
4. **교사 업무 과부하**: 개별 학생 관리의 어려움으로 인한 맞춤형 교육 제공의 제약

### 2.2 연구 필요성

#### 2.2.1 교육학적 필요성

- **Vygotsky의 근접발달영역(ZPD) 이론**: 각 학생의 현재 수준과 잠재 수준 사이의 최적 학습 지점 제공
- **Bloom의 분류학**: 지식, 이해, 적용, 분석, 종합, 평가 단계별 맞춤형 학습 지원
- **인지부하 이론**: 학생의 인지적 처리 능력을 고려한 학습 내용 제시

#### 2.2.2 기술적 필요성

- **AI 기술의 발전**: 대규모 언어 모델과 온톨로지 기술을 활용한 지식 표현 및 추론
- **빅데이터 분석**: 학습 행동 데이터의 실시간 수집 및 분석 기술
- **자동화 시스템**: 반복적이고 체계적인 학습 지원 프로세스의 자동화

#### 2.2.3 사회적 필요성

- **교육 격차 해소**: 개인 맞춤형 교육을 통한 교육 기회의 균등화
- **학습 효과 향상**: 개인화된 학습 경험을 통한 학습 성취도 향상
- **교사 역량 강화**: AI 시스템을 통한 교사의 전문성 보완 및 업무 효율화

## 3. 연구 내용 및 방법

### 3.1 전체 시스템 구조

본 시스템은 **오케스트레이션 에이전트(Orchestrator)**를 중심으로 22개의 전문 에이전트가 협력하여 작동하는 구조입니다.

```
┌─────────────────────────────────────────┐
│   오케스트레이션 에이전트 (Orchestrator)  │
│   - 실시간 맥락 수집 및 통합              │
│   - 다음 최선행동(NBA) 결정               │
│   - 에이전트 호출 및 순서 관리            │
│   - 30분 주기 메타-평가/재계획            │
└──────────────┬──────────────────────────┘
               │
    ┌──────────┼──────────┐
    │          │          │
    ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐
│ 핵심   │ │ 분석   │ │ 지원   │
│에이전트│ │에이전트│ │에이전트│
│1,4,14  │ │7-13    │ │2,3,5,6 │
│18-21   │ │15-17   │ │        │
└────────┘ └────────┘ └────────┘
```

### 3.2 22개 전문 에이전트 상세

#### 3.2.1 핵심 에이전트 (Core Agents)

**Agent 01: 온보딩 (Onboarding)**
- **역할**: 신규/대상 학생의 기본 프로필과 학습 이력 수집
- **기능**: 
  - 온보딩 설문 16개 질문 처리 (인지/감정/행동 영역)
  - 학습 스타일 분석 (계산형/개념형/응용형)
  - 첫 수업 전략 도출
- **데이터**: `mdl_alt42o_onboarding`, `mdl_alt42o_learning_assessment_results`
- **온톨로지**: OnboardingContext, LearningContextIntegration, FirstClassDecisionModel

**Agent 04: 활동 유형 분석 (Problem Activity)**
- **역할**: 학생의 문제 풀이 활동 패턴 분석
- **기능**: 활동 유형 분류 및 학습 전략 제안

**Agent 14: 현재 위치 분석 (Current Position)**
- **역할**: 학생의 현재 학습 진도 및 위치 파악
- **기능**: 진도 분석, 약점 진단, 학습 경로 제안

**Agent 18-21: 실행 계층 (Execution Layer)**
- **Agent 18**: 시그니처 루틴 (Signature Routine)
- **Agent 19**: 상호작용 콘텐츠 (Interaction Content)
- **Agent 20**: 개입 준비 (Intervention Preparation)
- **Agent 21**: 개입 실행 (Intervention Execution)

#### 3.2.2 분석 에이전트 (Analysis Agents)

**Agent 07-13: 데이터 분석**
- **Agent 07**: 상호작용 타겟팅 (Interaction Targeting)
- **Agent 08**: 평온함 관리 (Calmness)
- **Agent 09**: 학습 관리 (Learning Management)
- **Agent 10**: 개념 노트 (Concept Notes)
- **Agent 11**: 문제 노트 (Problem Notes)
- **Agent 12**: 휴식 루틴 (Rest Routine)
- **Agent 13**: 학습 이탈 (Learning Dropout)

**Agent 15-17: 통합 계획**
- **Agent 15**: 문제 재정의 (Problem Redefinition)
- **Agent 16**: 상호작용 준비 (Interaction Preparation)
- **Agent 17**: 남은 활동 (Remaining Activities)

#### 3.2.3 지원 에이전트 (Support Agents)

**Agent 02**: 시험 일정 관리 (Exam Schedule)
**Agent 03**: 목표 분석 (Goals Analysis)
**Agent 05**: 학습 감정 (Learning Emotion)
**Agent 06**: 교사 피드백 (Teacher Feedback)

**Agent 22**: 모듈 개선 (Module Improvement)
- **역할**: 시스템 자체의 성능 개선 및 진화
- **기능**: 10단계 진화 로드맵을 통한 자율적 업그레이드

### 3.3 핵심 기술 요소

#### 3.3.1 온톨로지 기반 지식 표현

**구조**:
- **Context Layer**: 학습 맥락 정보 수집 및 표현
- **Decision Layer**: 의사결정 모델 및 전략 수립
- **Execution Layer**: 실행 계획 및 행동 정의

**표준**:
- JSON-LD 형식의 온톨로지 표현
- OWL 2.0 기반 클래스 및 속성 정의
- RDF 스키마를 활용한 관계 표현

**예시 (Agent 01)**:
```json
{
  "@id": "mk:OnboardingContext/instance_001",
  "@type": "mk:OnboardingContext",
  "mk:hasStudentGrade": "중2",
  "mk:hasMathConfidence": 4,
  "mk:hasMathLearningStyle": "mk:Conceptual"
}
```

#### 3.3.2 룰 엔진 시스템

**구조**:
- YAML 기반 규칙 정의 (`rules.yaml`)
- 시나리오 그룹별 규칙 분류
- 조건-행동(Condition-Action) 매핑

**평가 방식**:
1. **주기적 체크**: 30분 주기 하트비트 시스템
2. **이벤트 체크**: 사용자 행동 기반 실시간 평가
3. **상태 변화 체크**: 데이터 변경 시 영향받는 룰만 선별 평가

**예시**:
```yaml
S0_R2_academy_info_collection:
  condition:
    - student_grade: exists
    - academy_name: exists
  action:
    - collect_academy_info
    - assess_difficulty_level
  priority: high
```

#### 3.3.3 데이터베이스 구조

**주요 테이블 그룹**:

1. **alt42o_* (Onboarding)**: 온보딩 관련 데이터
   - `mdl_alt42o_onboarding`: 온보딩 기본 정보
   - `mdl_alt42o_learning_assessment_results`: 학습 평가 결과
   - `mdl_alt42o_onboarding_reports`: 온보딩 리포트

2. **alt42g_* (General)**: 일반 학습 데이터
   - `mdl_alt42g_learning_goals`: 학습 목표
   - `mdl_alt42g_learning_progress`: 학습 진도
   - `mdl_alt42g_emotion_surveys`: 감정 설문

3. **alt42t_* (Teacher)**: 교사 관련 데이터
   - `mdl_alt42t_exams`: 시험 정보
   - `mdl_alt42t_study_status`: 학습 상태

4. **alt42i_* (Interaction)**: 상호작용 데이터
   - `mdl_alt42i_math_patterns`: 수학 패턴
   - `mdl_alt42i_pattern_solutions`: 패턴 해결책

5. **alt42_* (Core)**: 핵심 시스템 데이터
   - `mdl_alt42_students`: 학생 정보
   - `mdl_alt42_learning_sessions`: 학습 세션
   - `mdl_alt42_interaction_logs`: 상호작용 로그
   - `mdl_alt42_agent_registry`: 에이전트 레지스트리

**총 테이블 수**: 200개 이상

#### 3.3.4 이벤트 기반 아키텍처

**이벤트 타입**:
- `learning.answer_wrong`: 오답 이벤트
- `learning.answer_correct`: 정답 이벤트
- `learning.session_start`: 학습 시작
- `learning.session_end`: 학습 종료
- `interaction.click`: 클릭 이벤트

**처리 흐름**:
```
사용자 행동
    ↓
이벤트 발생
    ↓
워크플로우 이벤트 프로세서
    ↓
관련 시나리오 선별 평가
    ↓
결과 반환 및 상태 업데이트
```

### 3.4 연구 방법론

#### 3.4.1 설계 방법론

- **UI-동형(iso-UI) 구조**: URL 경로와 폴더 구조의 1:1 매핑
- **레이어 분리**: View → JS → PHP Controller → Service → Model
- **메타데이터 동기화**: 모든 변경사항을 `ui.map.yaml`에 반영

#### 3.4.2 개발 방법론

- **점진적 개발**: MVP(Minimum Viable Product)부터 시작하여 단계적 확장
- **테스트 주도 개발**: 각 에이전트별 단위 테스트 및 통합 테스트
- **문서화 중심**: 모든 설계 결정을 문서화하여 유지보수성 확보

#### 3.4.3 평가 방법론

- **데이터 품질 지표**: 완전성(Completeness), 정확성(Accuracy), 일관성(Consistency)
- **성능 지표**: 응답 시간, 처리량, 에러율
- **사용자 만족도**: 교사 및 학생 피드백 수집 및 분석

#### 3.4.4 아키텍처 선택 근거 및 설계 철학

**핵심 설계 철학: "최고수준 선생님 사고체계의 90% 수준 달성"**

본 시스템은 단순한 기술적 구현을 넘어서, **현직 최고수준 수학 선생님의 사고체계를 모방**하여 AI 시스템으로 구현하는 것을 목표로 합니다. 이를 위해 다음과 같은 아키텍처 선택을 하였습니다.

**1. 에이전트 기반 아키텍처 선택 근거**

**대안 아키텍처 비교**:

| 아키텍처 패턴 | 장점 | 단점 | 선택 이유 |
|------------|------|------|----------|
| **모놀리식** | 구현 단순, 성능 우수 | 확장성 제한, 유지보수 어려움 | ❌ 교육 도메인의 복잡성 미반영 |
| **마이크로서비스** | 확장성 우수, 독립 배포 | 네트워크 오버헤드, 분산 트랜잭션 복잡 | ❌ 교육 도메인 특성상 과도한 분산 |
| **에이전트 기반** | 도메인 전문성 반영, 유연한 협업 | 복잡도 관리 필요 | ✅ **선택: 교육 도메인 특성에 최적** |
| **이벤트 기반** | 느슨한 결합, 확장성 | 이벤트 순서 보장 어려움 | ⚠️ 보조 패턴으로 활용 |

**에이전트 기반 아키텍처를 선택한 핵심 이유**:

1. **도메인 전문성 반영**: 각 에이전트가 특정 교육 영역(온보딩, 감정 관리, 학습 분석 등)에 전문성을 가짐 → 실제 선생님의 역할 분담과 유사
2. **점진적 확장성**: 새로운 교육 기능 추가 시 새 에이전트만 추가하면 됨 → 모놀리식 대비 확장 용이
3. **실시간 협업**: 에이전트 간 실시간 정보 공유 및 협업 → 선생님의 다차원적 판단 과정 모방
4. **자가개선 가능**: Agent 22를 통한 시스템 자체의 진화 → 지속적 학습 가능

**2. 온톨로지 기반 지식 표현 선택 근거**

**대안 지식 표현 방식 비교**:

| 방식 | 장점 | 단점 | 선택 이유 |
|------|------|------|----------|
| **관계형 DB만 사용** | 구현 단순, 성능 우수 | 의미론적 관계 표현 제한 | ❌ 복잡한 교육 지식 표현 불가 |
| **그래프 DB** | 관계 표현 우수 | 표준화 부족, 학습 곡선 | ⚠️ 고려했으나 표준화 중요 |
| **온톨로지 (OWL/JSON-LD)** | 표준화, 추론 가능, 재사용성 | 구현 복잡도 | ✅ **선택: 교육 지식의 구조화된 표현** |

**온톨로지를 선택한 핵심 이유**:

1. **표준화**: W3C 표준(OWL 2.0, JSON-LD) 준수 → 장기적 호환성 및 확장성 보장
2. **의미론적 추론**: 단순 데이터 저장이 아닌 의미론적 관계를 통한 자동 추론 가능
3. **재사용성**: 다른 교육 시스템과 온톨로지 공유 및 확장 가능
4. **검증 가능성**: 온톨로지 검증 도구를 통한 일관성 보장

**3. 룰 엔진 선택 근거**

**대안 의사결정 방식 비교**:

| 방식 | 장점 | 단점 | 선택 이유 |
|------|------|------|----------|
| **머신러닝만 사용** | 자동 학습, 패턴 인식 | 설명 불가능, 데이터 의존 | ⚠️ 보조로 활용 |
| **룰 엔진만 사용** | 설명 가능, 제어 용이 | 규칙 관리 복잡 | ✅ **선택: 교육 도메인의 명시적 지식** |
| **하이브리드** | 두 방식의 장점 결합 | 복잡도 증가 | 🔮 향후 진화 방향 |

**룰 엔진을 선택한 핵심 이유**:

1. **설명 가능성**: 교육 도메인에서 "왜 이 결정을 했는가"에 대한 설명이 필수적 → 룰 엔진은 완벽한 추적 가능성 제공
2. **도메인 전문가 지식 반영**: 교사의 노하우를 룰로 직접 표현 가능
3. **명시적 제어**: 교육 정책 및 규칙을 명시적으로 관리 가능
4. **점진적 개선**: 룰 추가/수정을 통한 점진적 시스템 개선

**4. 계층 구조 설계 근거 (Context → Decision → Execution)**

**설계 원칙**: "선생님의 사고 과정 모방"

```
실제 선생님의 사고 과정:
1. 상황 파악 (Context) → 학생의 현재 상태, 맥락 수집
2. 판단 및 결정 (Decision) → 최적 전략 수립
3. 실행 (Execution) → 구체적 행동 수행
```

**계층 분리의 핵심 이유**:

1. **관심사 분리**: 각 계층이 명확한 책임을 가짐 → 유지보수성 향상
2. **변경 영향 최소화**: 한 계층의 변경이 다른 계층에 미치는 영향 최소화
3. **테스트 용이성**: 각 계층을 독립적으로 테스트 가능
4. **재사용성**: Context나 Decision을 다른 시나리오에서 재사용 가능

**5. 강결합 구조의 의도적 선택 및 완화 전략**

**현재 구조의 특성**:
- 에이전트들이 데이터·온톨로지·룰 엔진에 직접 의존
- 이는 **의도적 설계**로, 초기 개발 속도와 성능을 우선시

**왜 강결합을 선택했는가**:
1. **개발 속도**: 초기 프로토타입 단계에서 빠른 개발 필요
2. **성능**: 직접 접근을 통한 낮은 레이턴시
3. **단순성**: 복잡한 추상화 레이어 없이 직관적 구조

**완화 전략 (단계적 적용)**:
1. **Phase 1 (현재)**: 강결합 구조로 빠른 개발 및 검증
2. **Phase 2 (3개월)**: 인터페이스 레이어 도입으로 의존성 역전
3. **Phase 3 (6개월)**: 이벤트 기반 메시지 버스로 완전한 느슨한 결합

**6. 확장성 보장 전략**

**현재 구조의 확장성 한계 인식**:
- 22개 에이전트는 현재 구조로 관리 가능
- 50개 이상 에이전트 시 구조 개선 필요

**확장성 보장을 위한 설계**:
1. **에이전트 레지스트리**: 동적 에이전트 등록 및 발견
2. **표준화된 인터페이스**: 모든 에이전트가 동일한 프로토콜 사용
3. **의존성 그래프 관리**: 자동 의존성 분석 및 실행 순서 결정 (개선 계획)
4. **버전 관리**: 에이전트별 버전 관리로 하위 호환성 보장

#### 9.1.1 에이전트 간 동기화 및 경합 조건

**취약점 요약**:
22개 에이전트가 규칙·이벤트·온톨로지 기준으로만 연결되어 있어, 동시 실행 시 Deadlock·중복 판단·경합 조건이 발생할 가능성이 높습니다. 특히 핵심 에이전트(Agent 01, 04, 14)와 실행 계층 에이전트(Agent 18-21)가 동일한 데이터 소스에 접근할 때 충돌이 발생할 수 있습니다.

**현재 대응 상태**:

- ✅ 권한 계층: `agents_rules.yaml`에 에이전트별 권한 레벨 정의 (Level 10: Agent 01, Level 8: Agent 14)

- ⚠️ 미완성: 동시 실행 시 락(Lock) 메커니즘 부재

**영향도**: 높음  
**발생 가능성**: 중간 (동시 사용자 100명 이상 시 증가)  
**우선순위**: P1 (높음)

**개선 계획**:

   - 순환 의존성 탐지 및 경고 시스템
2. **중기 (3개월)**: 분산 락 메커니즘 구현

3. **장기 (6개월)**: 완전한 트랜잭션 관리 시스템
   - 2단계 커밋(2PC) 또는 사가 패턴 적용
   - 롤백 메커니즘 구현

#### 9.1.2 룰 엔진 충돌 및 드리프트 문제

**취약점 요약**:
룰 엔진(1,000개 이상)이 우선순위·범위·충돌탐지 시스템 없이 운영되기 때문에 장기적으로 룰 드리프트(rule drift)가 거의 확실하게 발생합니다. 특히 시나리오 그룹(S0, S1, S2 등) 간 규칙이 중복되거나 상충할 때 자동 해결 메커니즘이 부족합니다.

**현재 대응 상태**:
- ✅ 우선순위 시스템: 각 룰에 priority 필드 존재 (1-100 범위)
- ✅ 충돌 해결: `ConflictResolver.php`에서 우선순위 기반 해결
- ⚠️ 미완성: 룰 범위(scope) 정의 부재
- ⚠️ 미완성: 룰 드리프트 자동 탐지 시스템 부재
- ⚠️ 미완성: 룰 버전 관리 시스템 부재

**영향도**: 중간  
**발생 가능성**: 높음 (시간 경과에 따라 증가)  
**우선순위**: P1 (높음)

**룰 엔진 생명주기 관리 전략 (Rule Lifecycle Management)**

**룰 생명주기 단계 정의**:

```
[1] Draft (초안)
    ↓
[2] Testing (테스트) → A/B 테스트
    ↓
[3] Active (활성) → 프로덕션 사용
    ↓
[4] Monitoring (모니터링) → 성능 추적
    ↓
[5] Deprecated (사용 중단) → 점진적 제거
    ↓
[6] Archived (보관) → 이력 관리
```

**각 단계별 관리 전략**:

**1. Draft 단계**:
- 룰 작성 후 자동 검증 (문법, 타입, 의존성)
- 충돌 사전 탐지 (기존 룰과의 충돌 검사)
- 범위(scope) 정의 필수: `agent_id`, `scenario_group`, `data_types`

**2. Testing 단계**:
- A/B 테스트 프레임워크를 통한 검증
- 제한된 사용자 그룹에만 적용
- 성능 지표 수집 (실행 빈도, 성공률, 응답 시간)

**3. Active 단계**:
- 프로덕션 환경에 배포
- 실시간 모니터링 (실행 빈도, 성공률, 에러율)
- 자동 알림 시스템 (이상 징후 탐지 시)

**4. Monitoring 단계**:
- 룰 드리프트 탐지 알고리즘 실행
- 사용 빈도 저하 탐지 (30일 이상 미사용 시 경고)
- 성능 저하 탐지 (성공률 20% 이상 하락 시 경고)

**5. Deprecated 단계**:
- 새 룰로 대체 준비
- 점진적 트래픽 감소 (10% → 50% → 100%)
- 롤백 가능성 유지

**6. Archived 단계**:
- 이력 보관 (Git 기반 버전 관리)
- 분석 및 학습 데이터로 활용

**룰 드리프트 탐지 알고리즘**:

```yaml
rule_drift_detection:
  metrics:
    - execution_frequency: "30일 평균 실행 빈도"
    - success_rate: "성공률 (목표: 80% 이상)"
    - response_time: "응답 시간 (목표: 500ms 이하)"
    - conflict_rate: "충돌 발생률 (목표: 5% 이하)"
  
  thresholds:
    low_usage: "30일 이상 실행 빈도 0"
    performance_degradation: "성공률 20% 이상 하락"
    high_conflict: "충돌률 10% 이상"
  
  actions:
    - alert: "관리자에게 알림"
    - deprecate: "자동 사용 중단 후보 지정"
    - archive: "6개월 이상 미사용 시 자동 보관"
```

**룰 버전 관리 시스템 설계**:

**버전 체계**: `MAJOR.MINOR.PATCH`
- **MAJOR**: 호환성 깨는 변경 (조건/행동 구조 변경)
- **MINOR**: 호환성 유지하는 기능 추가
- **PATCH**: 버그 수정

**버전 관리 프로세스**:
1. 룰 변경 시 자동 버전 증가
2. Git 기반 이력 관리
3. 롤백 기능 (이전 버전으로 즉시 복구)
4. 변경 이력 추적 (누가, 언제, 왜 변경했는지)

**A/B 테스트 프레임워크**:

**테스트 설계**:
- **대조군**: 기존 룰 (A)
- **실험군**: 새 룰 (B)
- **할당 비율**: 50:50 (조정 가능)
- **테스트 기간**: 최소 7일 (통계적 유의성 확보)

**평가 지표**:
- 성공률 비교
- 응답 시간 비교
- 사용자 만족도 비교
- 비즈니스 지표 비교 (학습 성취도 등)

**개선 계획**:
1. **단기 (1개월)**: 룰 범위(scope) 정의 시스템 구축
   - 각 룰에 적용 범위(에이전트, 시나리오, 데이터 타입) 명시
   - 범위 기반 충돌 사전 탐지
   - 룰 메타데이터 스키마 확장
2. **중기 (3개월)**: 룰 드리프트 모니터링 시스템
   - 룰 실행 빈도 및 성공률 추적
   - 사용되지 않는 룰 자동 탐지 및 알림
   - 룰 생명주기 상태 관리 시스템
3. **장기 (6개월)**: 룰 버전 관리 및 A/B 테스트 시스템
   - 룰 버전 관리 시스템 구축 (Git 기반)
   - A/B 테스트 프레임워크 구현
   - 자동화된 룰 배포 파이프라인

#### 9.1.3 온톨로지 변경의 연쇄 효과(Cascade Side Effect)

**취약점 요약**:
온톨로지가 실행 레이어와 직접 연결되어 있어, 변경 시 전체 시스템에 Cascade Side Effect가 발생할 구조입니다. 특히 Context Layer의 변경이 Decision Layer와 Execution Layer에 즉시 영향을 미치며, 이로 인한 예기치 못한 동작이 발생할 수 있습니다.

**현재 대응 상태**:
- ✅ 계층 구조: Context → Decision → Execution 계층 분리
- ✅ JSON-LD 표준: 표준화된 온톨로지 표현 형식 사용
- ⚠️ 미완성: 온톨로지 변경 영향도 분석 도구 부재
- ⚠️ 미완성: 버전 관리 및 롤백 메커니즘 부재
- ⚠️ 미완성: 의존성 그래프 시각화 도구 부재

**영향도**: 높음  
**발생 가능성**: 낮음 (온톨로지 변경 빈도가 낮음)  
**우선순위**: P2 (중간)

**개선 계획**:
1. **단기 (2개월)**: 온톨로지 변경 영향도 분석 도구 개발
   - 클래스 간 의존성 그래프 자동 생성
   - 변경 시 영향받는 에이전트 및 룰 자동 탐지
2. **중기 (4개월)**: 온톨로지 버전 관리 시스템
   - Git 기반 버전 관리
   - 변경 이력 추적 및 롤백 기능
3. **장기 (6개월)**: 온톨로지 변경 시뮬레이션 시스템
   - 변경 전 시뮬레이션을 통한 영향도 예측
   - 자동 테스트를 통한 검증

**Fail-Safe 시나리오 기반 실행 흐름**

**시나리오 1: 에이전트 실행 실패**

```
[정상 흐름]
에이전트 호출 → 실행 성공 → 결과 반환

[Fail-Safe 흐름]
에이전트 호출 → 실행 실패 감지
    ↓
[1단계] 재시도 (최대 3회, 지수 백오프)
    ↓ (재시도 실패 시)
[2단계] 대체 에이전트 호출 (동일 기능)
    ↓ (대체 에이전트 없음)
[3단계] 기본값 반환 (안전한 기본 전략)
    ↓
[4단계] 로깅 및 알림 (관리자에게 즉시 알림)
    ↓
[5단계] 자동 복구 시도 (30분 후 재시도)
```

**시나리오 2: 데이터베이스 연결 실패**

```
[정상 흐름]
DB 연결 → 쿼리 실행 → 결과 반환

[Fail-Safe 흐름]
DB 연결 실패 감지
    ↓
[1단계] 연결 풀 재시도 (최대 3회)
    ↓ (재시도 실패 시)
[2단계] 읽기 전용 복제본으로 전환
    ↓ (복제본 없음)
[3단계] 캐시된 데이터 반환 (최대 5분 전 데이터)
    ↓
[4단계] 큐에 작업 저장 (나중에 재처리)
    ↓
[5단계] 알림 및 모니터링 (DB 팀에 즉시 알림)
```

**시나리오 3: 룰 엔진 충돌**

```
[정상 흐름]
룰 평가 → 단일 룰 매칭 → 행동 실행

[Fail-Safe 흐름]
여러 룰 충돌 감지
    ↓
[1단계] 우선순위 기반 해결 (ConflictResolver)
    ↓ (우선순위 동일)
[2단계] 신뢰도(confidence) 기반 해결
    ↓ (신뢰도 동일)
[3단계] 최근 성공률 기반 해결
    ↓ (해결 불가)
```

### 9.5 젠가 블록 분석 (핵심 의존성 분석)

**젠가 블록 개념**: 시스템에서 제거 시 전체 시스템이 무너지는 핵심 구성 요소

#### 9.5.1 Tier 1: 절대 핵심 블록 (제거 불가)

**1. 오케스트레이션 에이전트 (Orchestrator)**
- **의존성**: 모든 에이전트가 Orchestrator를 통해 호출됨
- **제거 시 영향**: 전체 시스템 중단
- **대체 방안**: 없음 (단일 장애점)
- **개선 전략**: 고가용성(HA) 구성 필요

**2. Agent 01 (온보딩)**
- **의존성**: 다른 모든 에이전트가 온보딩 데이터에 의존
- **제거 시 영향**: 학생 프로필 생성 불가 → 전체 시스템 기능 정지
- **대체 방안**: 없음
- **개선 전략**: 데이터 검증 강화, 백업 메커니즘

**3. 데이터베이스 (MySQL)**
- **의존성**: 모든 에이전트가 DB에 직접 접근
- **제거 시 영향**: 데이터 저장/조회 불가 → 전체 시스템 중단
- **대체 방안**: 읽기 전용 복제본 (부분적 완화)
- **개선 전략**: DB 클러스터링, 자동 백업

#### 9.5.2 Tier 2: 높은 의존성 블록 (제거 시 주요 기능 손실)

**4. 룰 엔진 (Rule Engine)**
- **의존성**: 모든 에이전트가 룰 엔진을 통해 의사결정
- **제거 시 영향**: 의사결정 불가 → 기본값만 반환 가능
- **대체 방안**: 하드코딩된 기본 전략 (제한적)
- **개선 전략**: 룰 엔진 고가용성 구성

**5. 온톨로지 시스템**
- **의존성**: Context/Decision/Execution 레이어가 온톨로지 기반
- **제거 시 영향**: 구조화된 지식 표현 불가 → 시스템 기능 대폭 저하
- **대체 방안**: 관계형 DB로 대체 (기능 제한)
- **개선 전략**: 온톨로지 버전 관리, 롤백 메커니즘

**6. Agent 14 (현재 위치 분석)**
- **의존성**: 실행 계층 에이전트들이 현재 위치 정보 필요
- **제거 시 영향**: 학습 경로 추천 불가
- **대체 방안**: Agent 01 데이터로 대체 (정확도 저하)
- **개선 전략**: 데이터 중복 저장, 캐싱

#### 9.5.3 Tier 3: 중간 의존성 블록 (제거 시 부분 기능 손실)

**7. Agent 18-21 (실행 계층)**
- **의존성**: 실제 개입 실행 담당
- **제거 시 영향**: 개입 실행 불가 → 분석만 가능
- **대체 방안**: 수동 실행 (교사 개입)
- **개선 전략**: 트랜잭션 관리, 롤백

**8. 이벤트 시스템**
- **의존성**: 실시간 반응을 위한 이벤트 처리
- **제거 시 영향**: 실시간 반응 불가 → 주기적 체크만 가능
- **대체 방안**: 하트비트 시스템만 사용
- **개선 전략**: 이벤트 큐 고가용성

2. **온톨로지 기반 지식 표현**: 교육 지식의 구조화된 표현 및 재사용성 확보
3. **룰 엔진**: 교육 도메인에서 필수적인 설명 가능성(Explainability) 제공
4. **계층 구조**: 선생님의 사고 과정(Context → Decision → Execution) 모방

**강결합 구조의 의도적 선택**: 초기 개발 속도와 성능을 우선시한 전략적 선택이며, 단계적 완화 계획을 수립하여 장기적 확장성을 보장합니다.

### 10.3 시스템 안정성 및 확장성 보장
