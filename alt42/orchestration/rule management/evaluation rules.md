좋아 👍 이해했어.
이번에는 **다른 에이전트의 룰셋을 평가할 때 재사용 가능한 “표준 평가 템플릿”** 으로 제시해줄게.
즉, **구조는 유지하되 내용은 비워둔 형태**야 — 평가할 룰셋을 넣으면 바로 채점표로 쓸 수 있게 설계했어.

## 🧩 1️⃣ 구조적 완성도 (Structure Integrity)

| 점검 항목                | 평가 기준                               | 상태(✅⚠️❌) | 메모 |
| -------------------- | ----------------------------------- | -------- | -- |
| 룰 ID와 Description 존재 | 모든 rule_id에 설명이 포함되어 있는가            |          |    |
| Priority 중복 없음       | 동일 priority 간 우선순위 충돌이 없는가          |          |    |
| Conditions 명시        | 모든 룰에 최소 1개 이상의 condition이 정의되어 있는가 |          |    |
| Action 정의            | 모든 룰에 action 블록이 존재하는가              |          |    |
| Confidence 값 존재      | 0~1 범위 내 confidence가 정의되어 있는가       |          |    |
| Rationale 존재         | 룰별 근거가 명시되어 있는가                     |          |    |

## 🧠 2️⃣ 논리적 일관성 (Logic Coherence)

| 점검 항목             | 평가 기준                                                                | 상태(✅⚠️❌) | 메모 |
| ----------------- | -------------------------------------------------------------------- | -------- | -- |
| Condition 필드명 일관성 | 동일 의미의 필드명이 일관되게 사용되는가                                               |          |    |
| Operator 일관성      | 비교 연산자 사용이 표준화되어 있는가 (`==`, `<`, `>=`, `in`, `matches`)              |          |    |
| Action 타입 일관성     | `recommend_path`, `analyze`, `alert`, `display_message` 등의 사용이 일관적인가 |          |    |
| 중복 의미 룰 없음        | 유사 조건/행동을 가진 룰이 중복되어 있지 않은가                                          |          |    |
| Default Rule 존재   | fallback 상황을 처리할 기본 룰이 정의되어 있는가                                      |          |    |
| 시나리오 위계 구조        | 룰이 단계(S1~S5 등)별로 논리적 계층을 유지하는가                                       |          |    |

## 🔗 3️⃣ 데이터 연결성 (Data Connectivity)

| 점검 항목                  | 평가 기준                                     | 상태(✅⚠️❌) | 메모 |
| ---------------------- | ----------------------------------------- | -------- | -- |
| 입력 필드 정의 명확            | 입력 데이터의 출처(field)가 명확히 정의되어 있는가           |          |    |
| 출력 Action 경로 명확        | 출력 결과가 다른 에이전트나 모듈로 전달 가능한가               |          |    |
| Null 체크 포함             | `"!=": null` 조건 등 데이터 안전성이 확보되어 있는가       |          |    |
| 데이터 스키마 충돌 없음          | 동일 필드가 일관된 데이터 타입으로 사용되는가                 |          |    |
| Derived Description 생성 | `generate_description` 결과가 후속 분석에 활용 가능한가 |          |    |

## ⚙️ 4️⃣ 실행 검증 가능성 (Operational Verifiability)

| 점검 항목               | 평가 기준                               | 상태(✅⚠️❌) | 메모 |
| ------------------- | ----------------------------------- | -------- | -- |
| Display Message 명확성 | 사용자에게 전달되는 메시지가 구체적이고 명확한가          |          |    |
| Priority 흐름 자연성     | 우선순위 값이 상위(높음) → 하위(낮음) 흐름으로 자연스러운가 |          |    |
| 룰 간 충돌 최소화          | 동일 조건 내 상충되는 Action이 없는가            |          |    |
| Default Rule 동작 검증  | 조건 불일치 시 fallback 메시지가 반환되는가        |          |    |
| Confidence 분포 균형    | 모든 룰의 confidence가 균형된 범위에 분포하는가     |          |    |
| 로그 추적 가능성           | 각 룰이 로그나 기록 기반 검증이 가능한가             |          |    |
