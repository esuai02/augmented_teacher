## 1. 온톨로지란 무엇인가?

### 쉬운 설명

**온톨로지 = 수학교육 지식의 체계적 정리**

비유: 도서관의 분류 시스템
- 책(개념)들이 어떤 관계에 있는지 정의
- 어떤 책을 먼저 읽어야 하는지 (선수학습)
- 어떤 책들이 비슷한지 (유사개념)
- 어떤 주제로 묶이는지 (단원체계)

### Mathking에서 온톨로지의 역할

```
선생님의 지식 → 온톨로지로 구조화 → 22개 에이전트가 활용 → 학생 맞춤 지도
```

**예시**:
- **선생님**: "이차함수는 일차함수를 먼저 알아야 해"
- **온톨로지**: `이차함수 --선수학습--> 일차함수`
- **Agent04 (문제활동)**: 일차함수 부족 학생에게 이차함수 문제 미추천

## 2. 왜 온톨로지가 필요한가?

### 문제 상황

❌ **온톨로지 없을 때**:
- AI가 학생에게 어려운 문제를 무작위로 추천
- 학생 유형을 고려하지 못한 획일적 피드백
- 선생님의 교육철학이 시스템에 반영 안됨
- 매번 같은 질문에 다른 답변 (일관성 부족)

✅ **온톨로지 있을 때**:
- 학생 수준에 맞는 문제 자동 추천
- 학생 유형별 맞춤 피드백
- 선생님의 교육방침이 시스템 전체에 적용
- 일관된 기준으로 학생 지도

### 22개 에이전트가 온톨로지를 사용하는 예

| 에이전트 | 온톨로지 활용 |
|---------|-------------|
| Agent03 (목표분석) | 교육과정 온톨로지 → 학년별 성취기준 매핑 |
| Agent04 (문제활동) | 문제유형 온톨로지 → 난이도별 문제 추천 |
| Agent05 (학습감정) | 페르소나 온톨로지 → 학생 유형별 감정 관리 |
| Agent07 (상호작용 타게팅) | 시나리오 온톨로지 → 유형별 개입 전략 선택 |
| Agent22 (개선 제안) | 협업 온톨로지 → 에이전트 간 협력 패턴 분석 |

## 3. 6개 온톨로지 레이어 개요

Mathking 시스템은 **6개의 온톨로지 레이어**로 구성됩니다.

### Layer 3.1: Agent/Task Collaboration Ontology
**에이전트 협업 패턴**

- **무엇을**: 22개 에이전트가 어떻게 협력하는지 정의
- **예시**: "성적 하락 학생 → Agent04(진도분석) + Agent09(학습관리) + Agent15(문제재정의) 협업"
- **선생님 역할**: 협업이 필요한 상황 정의 (예: 성적하락, 이탈징후)

### Layer 3.2: LMS Activities/Features Mapping Ontology
**Moodle LMS 연동 매핑**

- **무엇을**: Mathking 에이전트가 Moodle의 어떤 기능을 사용하는지 매핑
- **예시**: "Agent04(문제활동) → Moodle 퀴즈 모듈, 문제은행"
- **선생님 역할**: 현재 사용 중인 Moodle 활동 목록 제공

### Layer 3.3: Heartbeat-Based Dynamic Interaction Ontology
**시간 기반 상호작용**

- **무엇을**: 언제 학생에게 개입할지 시간 규칙 정의
- **예시**: "수업 시작 30분 전 → Agent02(시험일정) 알림"
- **선생님 역할**: 개입 타이밍 규칙 설정 (수업 전/중/후)

### Layer 3.4: Persona Correlation Ontology
**학생 유형 분류**

- **무엇을**: 학생 행동 유형(페르소나) 정의
- **예시**:
  - P_avoidant (회피형): 어려운 문제 회피
  - P_perfectionist (완벽주의형): 실수 두려워함
  - P_consistent (꾸준형): 규칙적 학습
- **선생님 역할**: 학생 유형 특성 정의 및 식별 기준 설정

### Layer 3.5: Persona Response Scenario Ontology
**유형별 대응 시나리오**

- **무엇을**: 학생 유형별로 어떻게 지도할지 전략 정의
- **예시**: "회피형 학생 → 쉬운 문제부터 시작, 성공 경험 제공"
- **선생님 역할**: 유형별 교수학습 전략 작성

### Layer 3.6: Content System Correlation Ontology
**콘텐츠 자동 생성**

- **무엇을**: 학습 콘텐츠 간 관계 및 자동 생성 규칙
- **예시**: "연습문제 없으면 → GPT로 유사문제 자동 생성"
- **선생님 역할**: 콘텐츠 생성 규칙 및 품질 기준 설정

## 4. 선생님의 수업 과정과 온톨로지 연결

### 실제 수업 과정 매핑

| 수업 단계 | 선생님의 활동 | 관련 온톨로지 | 관련 에이전트 |
|----------|-------------|-------------|------------|
| **1. 수업 준비** | 단원 계획, 학습목표 설정 | Layer 3.2 (LMS 매핑)<br>Layer 3.6 (콘텐츠) | Agent01 (온보딩)<br>Agent03 (목표분석) |
| **2. 학생 진단** | 선수학습 확인, 수준 파악 | Layer 3.4 (페르소나)<br>Layer 3.1 (협업) | Agent02 (시험일정)<br>Agent14 (현재위치) |
| **3. 수업 진행** | 개념 설명, 예제 풀이 | Layer 3.6 (콘텐츠)<br>Layer 3.5 (시나리오) | Agent04 (문제활동)<br>Agent10 (개념노트) |
| **4. 형성평가** | 이해도 체크, 오개념 파악 | Layer 3.4 (페르소나)<br>Layer 3.1 (협업) | Agent05 (학습감정)<br>Agent15 (문제재정의) |
| **5. 개별 지도** | 보충/심화 학습 | Layer 3.5 (시나리오)<br>Layer 3.3 (Heartbeat) | Agent07 (상호작용 타게팅)<br>Agent08 (침착도) |
| **6. 피드백** | 학습 결과 분석, 차시 계획 | Layer 3.1 (협업)<br>Layer 3.6 (콘텐츠) | Agent06 (선생님 피드백)<br>Agent22 (개선 제안) |

### 하루 수업 시나리오 예시

**중2 일차함수 수업 (3차시)**

```
🕐 수업 전 (30분 전)
├─ Agent02 (시험일정) → Layer 3.3 활용
│  └─ "오늘 일차함수 쪽지시험" 알림
│
🕑 수업 시작 (도입 10분)
├─ Agent01 (온보딩) → Layer 3.2 활용
│  └─ Moodle 출석체크, 학습목표 제시
│
🕒 수업 중 (전개 30분)
├─ Agent04 (문제활동) → Layer 3.6 활용
│  ├─ 기본문제 풀이
│  └─ 부족한 학생 → 자동 보충문제 생성
├─ Agent05 (학습감정) → Layer 3.4 활용
│  └─ 회피형 학생 감지 → Agent07 호출
│
🕓 수업 후 (정리 10분)
├─ Agent14 (현재위치) → Layer 3.1 활용
│  └─ 학습 진도 평가
└─ Agent22 (개선 제안) → Layer 3.1 활용
   └─ 다음 차시 개선 방향 제시
```

## 5. 온톨로지 구축 순서 (권장)

### 🎯 Phase 1: 기초 온톨로지 (1-2주)

#### Step 1.1: 교육과정 온톨로지 구축 (Layer 3.6 일부)
**왜 먼저**: 모든 학습의 기반이 되는 교육과정 체계

✅ **구축 내용**:
- 학년별 단원 구조
- 단원 간 선수학습 관계
- 핵심 개념 정의

📝 **GPT 프롬프트 예시**:
```
당신은 중학교 수학 교육과정 전문가입니다.
중학교 1학년 수학 교육과정을 다음 형식으로 정리해주세요:

1. 대단원명
2. 중단원명
3. 핵심개념 3-5개
4. 선수학습 개념 (있다면)
5. 후속학습 개념 (있다면)

예시:
- 대단원: 일차방정식
- 중단원: 일차방정식의 풀이
- 핵심개념: 등식의 성질, 이항, 계수 처리
- 선수학습: 정수의 사칙연산, 문자와 식
- 후속학습: 연립방정식
```

🎓 **선생님 작업**:
1. GPT로 초안 생성
2. 본인 학교 교육과정에 맞게 수정
3. 실제 수업 순서 반영

#### Step 1.2: 학생 유형 온톨로지 구축 (Layer 3.4)
**왜 필요**: 학생별 맞춤 지도의 핵심

✅ **구축 내용**:
- 5-7가지 학생 유형(페르소나) 정의
- 각 유형의 특성 3-5개
- 유형 식별 기준

📝 **GPT 프롬프트 예시**:
```
수학 학습에서 나타나는 학생 유형을 분류하려고 합니다.
다음 관점에서 5가지 유형을 제안해주세요:

1. 학습 태도 (적극적/소극적)
2. 문제 접근 방식 (신중/충동적)
3. 어려움 대처 (회피/도전)
4. 학습 속도 (빠름/보통/느림)
5. 자기조절 능력 (높음/낮음)

각 유형별로:
- 유형명 (한글+영문코드)
- 주요 특징 3개
- 식별 가능한 행동 패턴 3개
- 장점 2개, 단점 2개
```

🎓 **선생님 작업**:
1. GPT 제안 검토
2. 실제 학생 사례 대입하여 검증
3. 필요시 유형 추가/수정

✅ **구축 내용**:

- 상황별 개입 방법

📝 **GPT 프롬프트 예시**:
```
[회피형 학생] 유형을 위한 수학 지도 전략을 작성해주세요.

상황:
1. 어려운 문제를 만났을 때
2. 실수를 했을 때
3. 시험 전 불안할 때
4. 학습 동기가 떨어졌을 때

각 상황별로:
- 선생님의 초기 대응 (첫 30초)
- 구체적 개입 방법 (단계별)
- 사용할 수 있는 질문/멘트 예시
- 피해야 할 말/행동
```

🎓 **선생님 작업**:
1. 본인의 교육철학 반영
2. 효과적이었던 실제 사례 추가
3. 학교/학급 특성 고려

#### Step 2.2: 문제 유형 온톨로지 구축 (Layer 3.6)
**왜 필요**: 적절한 문제 추천 및 생성

✅ **구축 내용**:
- 문제 유형 분류 (개념이해/계산/응용/심화)
- 난이도 기준 (하/중/상)
- 유형별 학습 효과

📝 **GPT 프롬프트 예시**:
```

다음 관점에서:
1. 학습 목표별 분류 (개념 확인, 계산 연습, 응용, 창의)
2. 난이도별 기준
   - 하: 교과서 기본 문제 수준
   - 중: 교과서 응용/심화 문제 수준
   - 상: 경시대회 기초 수준
3. 각 유형별 대표 문제 예시 1개
4. 이 유형이 중요한 이유

추가로, 각 유형별로:
- 선수학습으로 필요한 개념
- 예상 소요시간
- 자주 나타나는 오류
```

🎓 **선생님 작업**:
1. 학교 교재/문제집 기준 통일
2. 실제 학생 오답 패턴 반영
3. 내신/모의고사 출제 경향 반영

### 🎯 Phase 3: 통합 온톨로지 (2-3주)

#### Step 3.1: 에이전트 협업 패턴 구축 (Layer 3.1)
**왜 필요**: 복합적 상황에서 여러 에이전트 협력

✅ **구축 내용**:
- 협업이 필요한 상황 정의
- 상황별 에이전트 조합
- 협업 순서 및 데이터 흐름

📝 **GPT 프롬프트 예시**:
```
다음 학생 상황에서 Mathking 에이전트들이 어떻게 협업해야 할지 설계해주세요.

상황: "수학 성적이 지난 달 대비 15% 하락한 중2 학생"

사용 가능한 에이전트 (22개):
1. Agent01: 온보딩
2. Agent02: 시험일정 식별
... (중략)
22. Agent22: 모듈성능 개선 제안

다음 형식으로 협업 패턴을 제시해주세요:
1. 상황 진단 단계 (어떤 에이전트가 먼저 분석?)
2. 원인 파악 단계 (어떤 데이터를 수집?)
3. 해결 방안 제시 단계 (어떤 에이전트가 협력?)
4. 실행 및 모니터링 단계
5. 각 단계별 소요 시간 및 우선순위
```

🎓 **선생님 작업**:
1. 실제 발생 빈도 높은 상황부터 작성
2. 학교 지도 프로세스와 일치시키기
3. 에이전트 협업 효과 검증

#### Step 3.2: Moodle LMS 매핑 (Layer 3.2)
**왜 필요**: 기존 LMS와 Mathking 연동

✅ **구축 내용**:
- 현재 사용 중인 Moodle 활동 목록
- 각 활동과 에이전트 매핑
- 데이터 연동 규칙

📝 **GPT 프롬프트 예시**:

## 6. GPT 활용 온톨로지 구축 방법

#### 원칙 2: 반복 개선
```
GPT 초안 생성 → 선생님 검토 → 수정 요청 → 재생성 → 최종 확정
```


### GPT 프롬프트 템플릿 모음

#### 템플릿 1: 개념 관계 정의

```
[핵심개념]: {개념 리스트}

다음 형식으로 개념 간 관계를 정의해주세요:
- 개념A --선수학습--> 개념B
- 개념B --유사개념--> 개념C
- 개념C --응용--> 개념D

각 관계에 대해:
1. 왜 이 관계가 중요한지
2. 관계의 강도 (필수/권장/참고)
3. 교수학습 시 주의사항
```

#### 템플릿 2: 학생 유형별 전략
```
[학생유형]: {유형명}
[특징]: {주요 특징 3개}

다음 상황에서 이 유형 학생을 지도하는 방법:

   - 예상 효과

2. [상황B]
   ...
```

#### 템플릿 3: 에이전트 협업 설계
```

[현재 데이터]: {학생 성적, 출석, 제출률 등}

다음 형식으로 협업 시나리오를 작성해주세요:

Phase 1: 진단
- 담당 에이전트: [에이전트명]
- 분석 내용: [무엇을 분석?]
- 산출물: [어떤 데이터 생성?]

Phase 2: 계획
- 담당 에이전트: [에이전트명]
- 이전 단계 입력: [Phase 1 산출물]
- 산출물: [개입 계획]

Phase 3: 실행
...
```

## 부록: 자주 묻는 질문 (FAQ)

### Q1: 온톨로지 구축에 시간이 얼마나 걸리나요?

**A**: 단계별 권장 시간
- Phase 1 (기초): 주당 2-3시간 × 2주 = 4-6시간
- Phase 2 (실행): 주당 3-4시간 × 3주 = 9-12시간
- Phase 3 (통합): 주당 2-3시간 × 3주 = 6-9시간
- **총 약 20-30시간** (방학 중 집중 작업 권장)

### Q2: GPT만으로 온톨로지를 다 만들 수 있나요?

**A**: GPT는 **80% 초안 생성**, 나머지 20%는 선생님의 **검증과 수정** 필요
- GPT 역할: 구조 제안, 예시 생성, 형식 변환
- 선생님 역할: 교육적 타당성 검증, 실제 수업 반영, 학교 특성 반영

### Q3: 다른 선생님과 온톨로지를 공유할 수 있나요?

**A**: 가능하며 권장함
- 기본 온톨로지 (교육과정, 문제유형) → 공유 후 학교별 커스터마이징
- 페르소나, 시나리오 → 학교/학급 특성 반영하여 개별 작성
- 협업 도구: GitHub, Google Drive, Notion 등 활용

### Q4: 온톨로지를 잘못 만들면 시스템이 오작동하나요?

**A**: 안전장치 있음
- **Layer 1 (기본)**: 잘못되면 시스템 전체 영향 → 신중하게
- **Layer 3 (확장)**: 잘못되면 해당 기능만 영향 → 실험적 수정 가능
- **검증 도구**: 온톨로지 일관성 자동 체크 기능 제공
- **롤백**: 이전 버전 복원 가능 (버전 관리 권장)

### Q5: 학기 중에 온톨로지를 수정할 수 있나요?

**A**: 가능하며, 수정 범위에 따라 다름
- **즉시 반영 가능**: Heartbeat 타이밍, 피드백 멘트
- **학기말 반영 권장**: 교육과정 구조, 에이전트 협업 패턴
- **학년 초 반영**: 학생 유형 분류 기준 (연속성 유지 위해)

### Q6: 온톨로지 없이 시스템을 사용할 수 있나요?

**A**: 제한적 사용 가능
- **기본 온톨로지** (시스템 제공) 사용 시: 일반적 기능만 작동
- **맞춤형 온톨로지** 없으면: 개인화 추천, 유형별 개입 불가
- **권장**: 최소한 Phase 1만이라도 구축 (2주 투자로 큰 효과)
